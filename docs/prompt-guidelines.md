# Prompt 编写规范

## 1. 目标

让团队在使用 `generate_page_from_prompt` 和 `update_page_by_instruction` 时，尽量得到稳定、可渲染、易校验的结果。

## 2. 推荐结构

建议一个 Prompt 至少包含以下信息中的 3 项：

- 页面类型
- 页面目标
- 核心区块
- 关键字段或列
- 操作按钮
- 约束条件

## 3. 推荐写法

### Dashboard

模板：

`帮我生成一个 <业务主题> dashboard，包含 <KPI>、<列表/表格>、<筛选区块> 和 <操作按钮>`

示例：

`帮我生成一个销售 dashboard，包含营收、转化率、重点商机表格和区域筛选`

### Form

模板：

`帮我生成一个 <业务主题> 表单，包含 <字段1>、<字段2>、<字段3>，并保留提交和保存动作`

示例：

`帮我生成一个客户录入表单，包含客户名称、负责人、预算、优先级和备注`

### Table / List

模板：

`帮我生成一个 <业务主题> 列表页，包含 <搜索>、<筛选>、<列信息> 和 <操作按钮>`

示例：

`帮我生成一个订单列表页，包含搜索、状态筛选、负责人列和操作按钮`

## 4. 二次修改指令写法

推荐以“动作 + 目标”表达：

- `把标题改成客户运营列表`
- `增加搜索筛选和操作按钮`
- `补一个备注输入区`
- `增加一个 KPI 区块`

组合示例：

`把标题改成客户运营列表，并增加搜索筛选和操作按钮`

## 5. 当前适合的表达

当前 Sprint 1 更适合这些能力：

- dashboard / form / table-list
- KPI 区块
- 表格
- 搜索筛选
- 基础按钮动作
- 表单字段
- 说明文本

## 6. 当前不适合直接要求的能力

以下需求即使写进 Prompt，也可能只得到提示或需要人工补充：

- 分页
- 详情跳转
- modal / drawer
- 图表
- 多页路由
- 真实 API 数据绑定
- 复杂权限控制

## 7. 反例

不推荐：

- `帮我做一个超级完整的后台`
- `生成一个很酷的页面`
- `把所有交互都补齐`

问题：

- 页面类型不明确
- 区块和字段不明确
- 超出 Sprint 1 能力边界

## 8. 团队约定

- 优先明确页面类型
- 优先列出核心区块，不追求一句话覆盖全部业务
- 需要二次修改时，尽量一次只提 1 到 3 个改动点
- 超出当前组件白名单的需求，要在 Prompt 里明确为“后续人工补齐”
