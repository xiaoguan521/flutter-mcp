# Flutter MCP AI App Builder 产品路线图

## 1. 文档目标

本文件用于统一项目的产品方向、阶段目标和里程碑定义，帮助团队围绕同一个最终目标推进研发。

当前项目已经具备以下基础能力：

- Flutter Studio 可加载、编辑、预览页面 DSL
- MCP UI Server 可提供页面读取、保存、版本化能力
- 页面可通过 JSON DSL 渲染为可交互 UI
- Android `debug` APK 已具备基础打包能力

在此基础上，下一阶段的核心目标不再是“继续补壳”，而是逐步实现：

- AI 通过 MCP 理解应用结构和页面资源
- AI 通过 MCP 生成或修改页面 / 应用 DSL
- Studio 承载人工校对、微调、版本固化
- Runtime 负责稳定渲染和动作执行
- Build 层负责导出 Web / APK / Desktop 产物

## 2. 最终目标

最终目标是构建一个 `AI + MCP + Studio + Runtime + Build` 闭环系统，使用户可以通过自然语言驱动页面、应用和构建流程。

目标闭环如下：

1. 用户输入业务需求
2. AI 通过 MCP 读取当前组件能力、页面资源、应用结构
3. AI 生成或修改 `Page DSL` 与 `App Schema`
4. Studio 实时预览并支持人工调整
5. 页面 / 应用被保存为版本化资源
6. 系统导出可运行的 Web / APK / Desktop 应用

一句话定义本项目：

`一个通过 AI 调用 MCP 来构建页面与应用的可视化生成平台。`

## 3. 产品边界

### 3.1 当前推荐路径

推荐分三层推进：

- 先实现 `AI 生成页面`
- 再实现 `AI 编排多页应用`
- 最后实现 `AI 触发构建与导出`

### 3.2 当前不建议一开始追求的目标

以下方向不建议作为第一优先级：

- 直接让 AI 生成完整 Flutter 原生工程
- 一开始就追求正式发版级别的全平台交付
- 在组件协议不稳定前引入过多业务定制能力

原因：

- 当前项目更适合先稳定 DSL 与应用模型
- 页面和应用的结构化表达是后续 AI 可控生成的基础
- 过早进入“全工程生成”会明显放大不可控性和维护成本

## 4. 核心产品能力图

最终系统需要具备 5 类核心能力：

### 4.1 应用建模能力

- 定义应用级元数据
- 定义路由、导航、页面组织关系
- 定义主题、布局壳、权限和全局配置

### 4.2 页面生成能力

- 生成单页面 DSL
- 修改现有页面 DSL
- 基于模板快速派生业务页面
- 支持结构校验、组件校验、动作校验

### 4.3 人机协同能力

- AI 生成初稿
- 人工在 Studio 中微调
- 支持本地草稿与版本固化
- 支持对比、回滚、重新生成

### 4.4 运行与交互能力

- 运行时渲染页面 DSL
- 执行 state / tool / resource / data action
- 连接真实数据源与业务接口

### 4.5 构建与交付能力

- 构建 Flutter Web
- 构建 Android APK
- 提供 Electron / WebView 容器
- 最终具备正式发布所需能力

## 5. 分阶段路线图

## 阶段 0：基础平台打通

### 阶段目标

完成可编辑、可渲染、可保存、可打包的最小可运行平台。

### 当前状态

基本完成。

### 已有成果

- 页面 DSL 渲染链路已通
- Studio 页面编辑链路已通
- 服务端保存与版本化链路已通
- 样例页种子导入已通
- Android `debug` APK 已能输出

### 阶段出口标准

- 至少一个页面可完成“读取 -> 编辑 -> 预览 -> 固化 -> 再加载”
- 服务端具备稳定资源 URI 与版本 URI
- Android `debug` 产物可构建

## 阶段 1：AI 单页生成 MVP

### 阶段目标

实现“输入一句需求，生成一张页面，并在 Studio 中继续编辑”的闭环。

### 关键价值

这是项目第一次把 AI 真正放到主工作流中，验证 AI 构建页面的可用性。

### 交付内容

- Prompt -> Page DSL 生成能力
- 页面结构校验能力
- 组件白名单与组件描述能力
- 页面二次修改能力
- 生成结果直接进入 Studio 预览

### 核心用户场景

用户输入：

`帮我生成一个销售 dashboard，包含 KPI、趋势区块、任务表格和筛选栏`

系统输出：

- 一份可运行的页面 DSL
- 在 Studio 中直接可视化预览
- 用户可继续微调并保存

### 阶段出口标准

- AI 能稳定生成至少 3 类页面
  - dashboard
  - form
  - table/list
- 生成页面可直接被 runtime 渲染
- 生成页面可被保存为版本资源

## 阶段 2：AI 多页应用 MVP

### 阶段目标

从“生成页面”升级为“生成应用骨架”。

### 关键价值

只有应用级建模完成，项目才能真正迈向“AI 构建 App / 应用”。

### 交付内容

- `App Schema` 设计与存储
- 路由与导航模型
- 应用级主题与布局壳
- 页面集合管理
- 页面与应用版本关系管理

### 核心用户场景

用户输入：

`帮我生成一个后台管理应用，包含首页、订单列表、订单详情、商品管理和设置页`

系统输出：

- 一个应用 manifest
- 多个页面资源
- 路由与导航结构
- Studio 中可切换各页面继续编辑

### 阶段出口标准

- AI 能生成多页应用骨架
- 页面间跳转正常
- 应用结构可版本化保存

## 阶段 3：数据与业务动作接入

### 阶段目标

让 AI 生成的页面从静态演示进入“可连接业务数据”的阶段。

### 关键价值

这是从“界面生成器”迈向“业务应用生成器”的关键一步。

### 交付内容

- 数据源描述模型
- API / mock 数据绑定能力
- 表单提交动作
- 表格查询、分页、筛选能力
- 结果状态与错误状态表达

### 核心用户场景

用户输入：

`帮我生成一个客户列表页，支持搜索、分页、状态筛选和详情跳转`

系统输出：

- 列表页 DSL
- 查询参数绑定
- 表格数据绑定
- 详情跳转动作

### 阶段出口标准

- 页面可绑定 mock 数据和真实接口
- AI 能生成包含业务动作的页面
- Studio 中可调试主要交互流程

## 阶段 4：AI 一键应用构建 MVP

### 阶段目标

让 AI 不只生成 DSL，还能触发应用构建和导出。

### 关键价值

这是“构建页面”走向“构建应用产物”的关键阶段。

### 交付内容

- 构建任务标准化
- AI 触发 Web / Android debug 构建
- 构建结果回传
- 构建失败信息结构化
- 导出产物元信息

### 核心用户场景

用户输入：

`把当前应用打一个 Android debug 包给我验证`

系统输出：

- 调用构建工具链
- 返回构建状态与日志摘要
- 输出 APK 路径

### 阶段出口标准

- AI 可发起构建任务
- 构建结果可回传到 Studio 或 MCP 响应中
- 至少支持 Web 与 Android debug 两类导出

## 阶段 5：发布级能力与治理

### 阶段目标

把平台从 Demo / MVP 提升到可交付、可治理的状态。

### 关键价值

决定项目是否能进入实际业务落地。

### 交付内容

- Android release 签名能力
- Web / Electron / Android WebView 完整验证
- DSL Schema 校验体系
- 版本 diff 与回滚机制
- 审计日志与操作追踪
- 自动化测试体系

### 阶段出口标准

- 关键链路具备测试覆盖
- 版本可回滚
- release 构建流程清晰
- 多端交付链路可验证

## 6. 建议里程碑

| 里程碑 | 名称 | 目标 | 核心产出 |
| --- | --- | --- | --- |
| M1 | AI 单页生成 MVP | 一句话生成单页面 | 页面生成、校验、保存、预览 |
| M2 | AI 多页应用 MVP | 生成多页应用骨架 | App Schema、路由、导航、页面集合 |
| M3 | AI 业务页面 MVP | 页面接入真实业务数据 | 数据源、动作绑定、表单与表格流程 |
| M4 | AI 构建导出 MVP | AI 触发构建 | Web / Android debug 构建与日志回传 |
| M5 | 发布级能力 | 可交付与可治理 | release、测试、回滚、审计 |

## 7. 当前阶段建议

结合当前代码基础，最推荐的近期推进顺序如下：

1. 先完成 `AI 单页生成 MVP`
2. 紧接着设计并落地 `App Schema`
3. 再推进 `AI 多页应用 MVP`
4. 然后补数据绑定与业务动作
5. 最后把构建流程 MCP 化

## 8. 近期优先级建议

当前最值得优先投入的 5 项工作：

1. 定义 `App Schema`
2. 扩展 MCP tools，覆盖生成、修改、校验、装配、导出
3. 扩充组件协议与组件库
4. 打通 AI -> MCP -> Studio 的编辑闭环
5. 将构建流程纳入 MCP 工具体系

## 9. 成功标准

当项目发展到较成熟阶段时，应满足以下标准：

- AI 能理解“应用”而不只是“页面”
- AI 生成结果可被稳定渲染
- 人工能方便地修订和固化 AI 结果
- 页面和应用都可以版本化与回滚
- 构建产物可以被导出和验证
- 平台具备基本的测试、审计和治理能力

## 10. 下一步建议

建议按以下顺序立刻启动：

1. 输出 `App Schema` 初版设计文档
2. 定义第二批 MCP tools 清单
3. 设计 AI 单页生成的输入输出协议
4. 补齐表单、导航、弹窗、数据列表类组件协议
5. 规划构建任务的 MCP 接口
